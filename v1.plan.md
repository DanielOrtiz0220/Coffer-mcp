# Coffer-MCP V1 Build Roadmap

## Summary

Start with a thin, runnable foundation before touching Plaid or Auth0. The first milestone should prove: `pnpm install`, `pnpm db:migrate`, `pnpm dev`, a Hono server, environment validation, Postgres migrations, and basic tests all work. After that, build the app in vertical slices toward one complete private MCP flow.

## Stages

1. **Foundation and database**
   - Scaffold the Node/TypeScript/Hono app with `dev`, `build`, `test`, lint, and migration scripts.
   - Add strict environment validation for the V1 secrets listed in the docs.
   - Create Postgres migrations for `users`, `plaid_items`, `accounts`, `transactions`, and `sync_state`.
   - Add a health route, structured safe logging, and a test harness.
   - This is where we should start because every later feature depends on repeatable local setup and schema correctness.

2. **Auth and MCP shell**
   - Implement Auth0 JWT verification: issuer, audience, JWKS, expiry, scope, local user mapping, and allowlist checks.
   - Add MCP discovery metadata endpoints and the MCP HTTP endpoint.
   - Register the four V1 tool names with argument validation, but initially return deterministic placeholder responses from empty database reads.
   - Add tests for auth failure cases and tool argument validation.

3. **Plaid Link setup flow**
   - Build the private setup route with Auth0-protected access.
   - Add Plaid `link_token` creation, public-token exchange, AES-256-GCM access-token encryption, and encrypted item storage.
   - Create the allowlisted local user during successful setup login.
   - Keep the setup UI minimal: connection status, launch Plaid Link, sync now, latest sync result, and safe error display.

4. **Plaid sync and normalized reads**
   - Implement `/transactions/sync` using stored cursors.
   - Upsert accounts, apply added/modified/removed transactions, store sync state, and handle `has_more` pagination.
   - Preserve old cursors on sync failure and soft-mark removed transactions.
   - Add focused tests for Plaid mapping, transaction direction derivation, idempotency, cursor handling, and encryption boundaries.

5. **Real tool behavior and deployment readiness**
   - Replace placeholder tool handlers with Postgres-backed implementations for `get_overview`, `list_accounts`, `search_transactions`, and `summarize_transactions`.
   - Add lazy sync with the documented defaults: 6-hour freshness window and 8-second per-tool timeout.
   - Verify logs never include financial values, tool arguments, raw Plaid payloads, or token material.
   - Update onboarding docs with the actual run/debug commands and complete a Railway smoke test using `list_accounts` and `search_transactions`.

## Public Interfaces

- Keep the V1 tool names and response shapes from `docs/v1-spec/tools.md`.
- Keep required environment variables from `docs/v1-spec/privacy-and-security.md`.
- Add canonical scripts: `pnpm dev`, `pnpm build`, `pnpm test`, `pnpm db:migrate`, and likely `pnpm lint`.
- Expose setup routes, MCP routes, and OAuth metadata routes only as needed by the V1 docs.

## Test Plan

- Unit tests: env validation, AES token encryption/decryption, Auth0 verifier behavior, tool argument validation, Plaid response mapping.
- Integration tests: migrations against test Postgres, repository queries, tool reads against seeded data.
- Smoke tests: local setup route, manual sync, MCP `list_accounts`, MCP `search_transactions`.
- Security checks: confirm logs omit tool argument values, balances, merchant names, account masks, Plaid tokens, and ciphertext material.

## Assumptions

- Use `pnpm`, TypeScript, Hono, Postgres, Auth0, Plaid, and Railway as already specified.
- Build for the private single-user MVP only; do not pull in post-MVP SaaS scope.
- Prefer a fast end-to-end vertical slice over building a broad internal framework first.
- Defer dashboard, background workers, webhooks, exports, write tools, and public signup.
